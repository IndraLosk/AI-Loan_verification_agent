# AI-агент проверки пакетов документов для льготных кредитов

Сервис автоматизирует проверку комплектности документов, подаваемых в банк для получения льготного кредита.  
Он принимает пакет файлов, определяет их типы по имени файла, проверяет соответствие требованиям программы (федеральной или региональной), выявляет ошибки и предупреждения, сохраняет результат проверки в PostgreSQL и возвращает структурированный отчёт.

## Технологический стек

- **Python 3.11** – язык разработки  
- **FastAPI** – веб-фреймворк для построения REST API  
- **PostgreSQL** – реляционная база данных для хранения результатов проверок  
- **SQLAlchemy** – ORM для работы с БД  
- **Alembic** – инструмент для миграций схемы БД  
- **Pydantic** – валидация данных и сериализация  
- **Pytest** – тестирование (покрытие логики определения типов и статусов)  
- **Docker & Docker Compose** – контейнеризация и оркестрация  
- **Make** – автоматизация часто используемых команд

## Запуск проекта (через Docker)

### 1. Клонируйте репозиторий

```bash
git clone git@github.com:IndraLosk/AI-Loan_verification_agent.git
cd AI-Loan_verification_agent
```

### 2. Создайте файл `.env` (опционально)

```bash
cp .env.example .env
```

### 3. Запустите сервисы через Docker Compose

```bash
docker compose up -d
```

### 4. Проверьте, что всё работает

Откройте в браузере [http://localhost:8000/docs](http://localhost:8000/docs)

### 5. Остановка

```bash
docker compose down
```

## API Эндпоинты

### `POST /api/checks`

Загружает пакет документов и запускает проверку.

**Параметры запроса (multipart/form-data):**

| Поле      | Тип          | Обязательное | Описание                                         |
|-----------|--------------|--------------|--------------------------------------------------|
| `program` | `string`     | Да           | `federal` или `regional`                         |
| `files`   | `file[]`     | Да           | Один или несколько файлов (PDF, DOCX, JPG, PNG) |

**Правила валидации:**

- **Программа `federal`** требует наличия: договор, спецификация, счёт, акт/УПД.
- **Программа `regional`** требует наличия: договор, счёт, акт/УПД.
- Тип документа определяется по имени файла (регистронезависимо) по паттернам:
  - `договор|contract` → `contract`
  - `спецификация|specification|спец` → `specification`
  - `счет|счёт|invoice` → `invoice`
  - `акт|упд|act|acceptance` → `act`
- Если тип не распознан — выдается предупреждение (`warning`).
- Допустимые расширения: `.pdf`, `.docx`, `.jpg`, `.jpeg`, `.png`. Иные форматы → ошибка.
- Максимальный размер файла: **20 МБ**. Превышение → ошибка.

**Итоговый статус:**

- `approved` – нет критических ошибок (могут быть только предупреждения).
- `rejected` – есть хотя бы одна ошибка (отсутствие обязательного документа, неверный формат, превышение размера).
- `check_in_progress` – зарезервирован для асинхронных сценариев, в текущей версии не используется.

**Пример запроса (curl):**

```bash
curl -X POST http://localhost:8000/api/checks \
  -F "program=federal" \
  -F "files=@/path/to/договор.pdf" \
  -F "files=@/path/to/спецификация.pdf" \
  -F "files=@/path/to/счет.pdf" \
  -F "files=@/path/to/акт.pdf"
```

**Пример успешного ответа (HTTP 200):**

```json
{
  "check_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "status": "approved",
  "status_label": "Можно заявлять в банк",
  "reason": null,
  "issues": [],
  "documents": [
    {"name": "договор.pdf", "detected_type": "contract", "size_kb": 142.5},
    {"name": "спецификация.pdf", "detected_type": "specification", "size_kb": 96.2},
    {"name": "счет.pdf", "detected_type": "invoice", "size_kb": 210.1},
    {"name": "акт.pdf", "detected_type": "act", "size_kb": 78.0}
  ],
  "extracted": {},
  "checked_at": "2025-03-15T14:32:00Z"
}
```

**Пример ответа с ошибкой (HTTP 200, но статус `rejected`):**


### `GET /api/checks`

Возвращает список всех выполненных проверок (без деталей файлов).

**Ответ (HTTP 200):**

```json
[
  {
    "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "program": "federal",
    "status": "approved",
    "checked_at": "2025-03-15T14:32:00Z",
    "documents_count": 4
  },
  {
    "id": "b2c3d4e5-f6a7-8901-bcde-f23456789012",
    "program": "regional",
    "status": "rejected",
    "checked_at": "2025-03-15T14:35:00Z",
    "documents_count": 3
  }
]
```

---

### `GET /api/checks/{check_id}`

Возвращает полную информацию о конкретной проверке.

**Параметры пути:**

- `check_id` – UUID проверки.

**Ответ (HTTP 200):**  
Аналогичен ответу `POST /api/checks` для соответствующей проверки.

**Ошибки:**

- `404 Not Found` – проверка с указанным ID не найдена.

## Тестирование

Тесты написаны с использованием `pytest` и находятся в папке `tests/`. Они покрывают:

- Определение типа документа по имени файла (5+ кейсов).
- Логику формирования итогового статуса (approved / rejected).
- Валидацию комплектности для `federal` и `regional`.
- Проверки формата и размера файлов.

**Запуск тестов внутри контейнера:**

```bash
docker compose exec api pytest tests/ -v
```

**Через Makefile:**

```bash
make test
```

**Локально (без Docker):**

```bash
pip install -r requirements.txt
pytest tests/
```

### Версионность и хранение файлов

В текущей реализации **файлы не сохраняются** на диск – они обрабатываются в памяти. В продакшене предполагается интеграция с объектным хранилищем (S3) и реализация версионности. Данный сервис – лишь API-слой для проверки.

### Безопасность

- Разграничение прав доступа не реализовано в демо-версии, но может быть добавлено с использованием механизмов аутентификации (JWT, OAuth2) и ролей.
- Все запросы к БД выполняются через SQLAlchemy, что защищает от SQL-инъекций.
